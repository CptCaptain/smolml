# Experiment 0.1 — baseline harness smoke run

**Status:** harness validation (not a scientific result). **Researcher handoff**
for the docs-builder — plain markdown, intuition + what the plot shows.

## Why this run exists

Task 0.1 builds the foundation: a byte-level data pipeline, an honest FLOP
counter, a transformer baseline behind a plug-in model interface, a train loop
that stops at a fixed **training-FLOP** budget, and a leaderboard. This run is the
proof the pieces wire together end-to-end and that the metric is wired correctly —
**not** a claim about enwik8. The corpus here is the tiny bundled English sample
(~5 KB), so absolute bpb numbers are not comparable to the literature; only the
*shape* of the curves and the harness mechanics matter.

## Setup

- Corpus: bundled offline sample, deterministic 90/10 train/val split.
- Two transformer baselines (pre-norm, RMSNorm, RoPE, causal), **same FLOP budget
  5×10¹⁰ training FLOPs** (a ceiling — both stop at ≤ budget), **same fixed eval
  context (64 bytes)**, CPU, seed 0:
  - **d32** — `d_model=32, n_layers=2` → 32,928 params, 220 steps.
  - **d64** — `d_model=64, n_layers=3` → 164,288 params, 46 steps.
- Both started from the uninformed baseline of **8.0 bpb** (uniform over 256 bytes).

## Result (validation bpb vs cumulative training FLOPs)

| model | params | steps @ 5e10 FLOPs | final val bpb |
| --- | ---: | ---: | ---: |
| d32 | 32,928 | 220 | **3.37** |
| d64 | 164,288 | 46 | 3.82 |

Trajectory (val bpb at increasing FLOPs):

- d32: `8.0 → 4.86 → 4.23 → 3.98 → … → 3.37`
- d64: `8.0 → 4.38 → 3.90 → 3.82`

## What the plot shows

The leaderboard plot is **bpb (y) vs cumulative training FLOPs (x, log scale)**,
one descending curve per run. Both curves fall steeply from 8.0 and bend toward a
floor — the familiar diminishing-returns shape of a loss-vs-compute curve, in
miniature.

The instructive bit: **at equal FLOPs the smaller model (d32) is lower** (3.37 vs
3.82). Each d32 step is far cheaper, so the same budget buys ~5× more optimizer
steps, and on a tiny corpus that extra optimization wins over the bigger model's
per-step capacity. This is exactly why the metric is *per FLOP* and read off a
*curve*, not from a single model at a single size: "bigger" is not automatically
"better per FLOP," and a one-budget snapshot can mislead.

Caveat to visualize honestly: this is a 5 KB sample, so the bigger model is also
data-starved here. The lesson is methodological (how to read bpb-vs-FLOP curves),
not a finding about transformer scaling.

## Worth visualizing (for the docs-builder)

- The two **bpb-vs-FLOPs curves** on shared log-x axes, with the equal-FLOP
  vertical line at 5×10¹⁰ annotated to show d32 below d64 — the "lower curve wins"
  idea from [loss-per-FLOP & scaling laws](../concepts/loss-per-flop-and-scaling-laws.md).
- An interactive slider over the FLOP budget that reads off each model's bpb at
  that budget, making "who wins depends on the budget" tangible.
- Tie-in to [compression = prediction](../concepts/compression-equals-prediction.md):
  8.0 bpb is "no model"; every bit shaved off is a byte the model stopped being
  surprised by.

## Reproduce

```bash
uv run smolml train --data sample --d-model 32 --layers 2 --seq-len 64 \
    --eval-seq-len 64 --budget 5e10 --run-name baseline-d32
uv run smolml train --data sample --d-model 64 --layers 3 --seq-len 64 \
    --eval-seq-len 64 --budget 5e10 --run-name baseline-d64
uv run smolml leaderboard --runs-dir runs
```
